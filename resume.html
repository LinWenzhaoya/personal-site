项目经历
Coding Agent Runtime 设计拆解与原型验证 2026.03

【背景】系统研究了 Claude Code、Codex CLI这两个不同定位的 agent 产品的 runtime 架构，提炼出核心设计差异：Claude Code 以注意力管理为核心追求模型推理能力的极致发挥，Codex CLI 以安全优先构建渐进式信任体系。在此基础上搭建了一个最小 Claude Code 风格的 agent 原型，通过 A/B 对比实验验证关键设计决策的合理性。

【方案】原型包含完整的 Tool Loop、search_code等工具、contextManager和 trace 记录系统。构造了一个含 6 个文件的 workspace 并埋入跨文件 bug，设计了两组实验：实验一对比上下文压缩开启/关闭对 agent 行为的影响，实验二对比搜索工具过滤/不过滤对决策质量的影响。分别使用 moonshot-v1-8k 和 kimi-k2.5 两个不同能力的模型跑实验，每组各 3 次，通过 trace 分析行为差异。

【结果】实验一验证了上下文治理的必要性：弱模型压缩开启时成功率 67%（平均 8 步），关闭时 0%（agent 陷入死循环重复同一失败操作）；强模型 ON 组 3/3 成功，OFF 组仍出现模型幻觉。实验二发现模型能力和 harness 设计的分层关系：弱模型两组均卡在 write_patch 基础能力瓶颈上（成功率均 1/3），搜索过滤的价值尚未体现；强模型解决基础能力问题后差异显现——filtered 组 3/3 成功，unfiltered 组出现 1 次信息过载引发的幻觉。两组实验共同指向：模型能力和 harness 复杂度是跷跷板关系，harness 设计需根据模型能力分层——先保证基础工具可用性，再优化信息过滤策略。

模块一：利用Claude Code创建MCP召回评估自动化skill

【问题】产品侧每周需例行评估MCP召回情况，由于存在众多MCP且每个MCP需评超过50个query，目前评估为纯手动流程——数百条query需逐一查看label页面并人工判断，效率极低。

【方案】基于对MCP调用链路各阶段的理解，明确debug日志中各环节关键判别字段，设计自动化路径：批量获取debugkey → 自动抓取解析debug信息 → 输出结构化评估报告。向Claude Code明确各阶段判别逻辑，结合典型case校准边界条件，经2-3轮纠偏后判别逻辑达到100%正确，为提速评估效率，引入20线程高并发方案，整套流程沉淀为可复用skill。

【结果】评估效率从手动逐条提升至500 query/min。

模块二：利用Claude Code创建Query需求强度评估自动化skill

【问题】医疗MCP覆盖6个场景（挂号、找医生等），每条query需判断所属场景及需求强度以决定是否触发组件。团队已有人工标准和标注数据，但直接输入主流LLM评估效果均不理想。

【方案】根因在于人工标准是面向人的自然语言描述，非AI可执行逻辑。核心方法：让Claude Code学习人工标准+数百条标注样本后，生成结构化"AI可执行版判别标准"；以100条样本实测，重点审查判别理由而非仅看对错，思路偏差时逐条纠偏，2-3轮迭代后标准完全对齐。

【结果】产出覆盖6场景×3级需求强度的AI执行标准skill，判别准确度与人工完全对齐，部分case覆盖维度优于人工，省下大量外包人力做人工评判工作。

模块四：反转实验

【第一轮实验】5%流量反转关闭全部在线MCP，净化分析将MCP分为三类：正向6个、无增益47个、负向17个。发现时延指标异常：关闭MCP理论应降低首token延迟约400ms，实测仅30ms。排查定位：反转开关置于Reader层，前序Pipeline（MCPRank、Planner、McpSkills）仍在运行，导致时延测量失真约13倍。

【结果】百度健康等6个MCP被验证为正向增益；发现并修正实验设计缺陷，推动双组方案落地。
