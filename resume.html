项目经历
基于自研 Coding Agent 的多模型 Benchmark 实验与能力诊断 2026.04

【背景】自主搭建了claude code风格的coding agent原型（含完整tool loop、上下文压缩、trace记录系统），设计评测框架，接入opus4.6、Deepseek-V3.2等五个模型，重点考察模型在'表面报错与真实病因分离'场景下的根因诊断与bug修复能力——这是用户实际开发中最头疼的复杂问题。五个模型各跑五轮，通过trace记录分析任务完成情况。

【核心发现1：推理能力强 ≠ Agent 能力强】DS-V3.2 同底座 Chat 与 Reason 版本，诊断正确率均为5/5，但 Chat 任务完成率 4/5，Reason 仅 2/5。原因是Reason模型工具格式遵循频频出错，将写文件的参数放置在调用读取文件工具的参数中，导致工具调用频频失败，且模型自身由于推理链过长，输入被污染，模型未意识到调用失败，反而以为自己已修复任务，直接宣布任务完成。而Chat 在相同 harness 下零格式错误，任务完成出色。

【核心发现2：部分模型严重缺失任务执行意愿】GPT 5.4 诊断正确率4/5，但写文件工具调用零次，每次都在等用户确认——把自己定位成顾问而非执行者。这是对话场景下的行为模式被带入了 Agent 场景，在需要自主执行的 Agent 场景下，模型无法胜任工作。

【四类失败模式归纳】综合五大模型表现，归纳出四类典型失败：工具格式遵循失败、执行意愿缺失、探索策略失误（如 Opus 碎片化读取恰好错过关键代码行，随后在错误方向上浪费多步）、整体性幻觉（如GLM凭空编造bug和修复过程）。每类失败指向不同的模型能力短板，并据此提出对应的评测指标优化方向。

模块一：利用Claude Code创建MCP召回评估自动化skill

【背景】产品侧每周需例行评估MCP召回情况，现阶段存在众多MCP工具且单MCP工具需评估超50个query，评估为纯手动流程——数百条query需逐一查看label页面人工判别召回情况，评估效率低下，并影响召回优化节奏。

【方案】基于对MCP调用链路各阶段的理解，向agent明确debug日志各环节关键判别字段，并设计自动化评估路径：批量获取debugkey→ 自动抓取解析debug信息 → 使用判别规则脚本批量识别召回情况，最终与agent协同开发自动化评估工具，并引入20线程并发加速评估效率。

【结果】实现20秒完成300+query召回情况判别，相比人工效率提升约500倍，沉淀为可复用skill，被团队持续复用，加速评估-优化闭环节奏。
